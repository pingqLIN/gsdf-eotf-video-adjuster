# LumaLift Extension Design Reference

This file is the project design source for the Chrome extension control surface. It describes the intended visual language and the boundary between project requirements and external references.

## Design Role

LumaLift is a compact professional video-control panel, not a landing page. The interface should feel like a precise instrument surface for repeated visual inspection: dense, quiet, readable, and trustworthy.

The first-screen priority is operation:

- enable or bypass the EOTF/GSDF effect
- adjust target luminance, gamma compensation, and filter amount
- switch between Basic, Advanced, and Diagnostic tabs without losing context
- run the Diagnostic web-camera luminance probe as a rough visual meter
- open the project-owned continuous frequency resolution reference and live curve only through the upper-right side-panel control

## Reference Boundary

Stitch exports may be used only as visual-language reference. They are not a product specification.

Use Stitch for:

- typography density and numeric-readout hierarchy
- recessed wells, fine borders, and top-lit bevels
- compact icon buttons and segmented controls
- dark instrument-panel contrast rhythm
- spacing cadence for labels, sliders, and status rows

Do not use Stitch for:

- feature names or workflow requirements
- chart data, formula behavior, or GSDF model interpretation
- diagnostic-pattern content
- screenshot imagery as a correctness reference
- the reference/test image shown in Stitch; it is not the project continuous frequency resolution pattern

The current project implementation remains authoritative for the GSDF-QC pattern. The continuous frequency resolution, modulation, line-pair, grayscale, and gradient checks are generated by the local canvas code and must not be replaced by the incorrect Stitch reference image.

## Visual Language

The panel uses an instrument-control surface aesthetic:

- dark obsidian chassis rather than decorative glass
- 4px spacing base with compact but readable grouping
- 6px controls and 8px shell corners, avoiding pill-shaped controls except status indicators
- inset control wells for sliders, previews, and grouped controls
- subtle top highlights and low-contrast dividers instead of heavy shadows
- restrained silver active states, green only for active/system confirmation, cyan only for analytical data

Typography uses two roles:

- Bahnschrift for labels, headings, and compact UI controls
- Cascadia Mono for numeric values, units, legends, and diagnostic labels

Letter spacing should stay modest for interactive controls. Wide tracking is reserved for small instrument labels and not for body text.

## Interaction Rules

Icon-only buttons must remain square, stable, and discoverable through `title` or `aria-label`.

Segmented controls must read as one physical control, not unrelated buttons. Active segments use a metallic high-contrast fill; inactive segments remain recessed.

Sliders must feel like physical faders:

- recessed 6px track
- rectangular thumb
- clear min/max labels where useful
- fixed value column to avoid layout jitter

The full GSDF-QC view must stay content-accurate. Visual polish can refine framing, labels, and control chrome, but the pattern structure remains project-owned.

## Quality Checks

Before shipping a UI change:

- run the TypeScript/build checks for the extension
- verify the rendered panel in a browser, not just by reading code
- check Basic, Advanced, Diagnostic, and the side-panel open/closed states where the change affects panel structure
- for the camera luminance probe, keep camera data local and label results as rough estimates rather than colorimeter, DICOM, or CSDF validation
- confirm text does not collide with controls at compact width
- confirm dark and light themes keep visible button boundaries
